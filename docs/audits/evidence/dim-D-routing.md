# Dimension D Audit: Multi-Account Routing / Failover / Health

Baseline: HEAD `1f6da97`, package `v1.2.7`.

State machine note: no explicit routing state machine. Transitions are implicit across `lib/accounts.ts:534-545`, `lib/accounts.ts:699-839`, `index.ts:1532-1579`, `index.ts:1974-2152`, and `lib/request/request-resilience.ts:20-74`.

Race-condition verdict: partial mitigation only. Refresh dedupe is explicit in `lib/refresh-queue.ts:99-125`, but account selection, cursor mutation, and debounced persistence remain unsynchronized in `lib/accounts.ts:621-624`, `lib/accounts.ts:693-696`, and `lib/accounts.ts:1166-1187`.

5xx rotation path trace: primary request gets `5xx` at `index.ts:1974-2064`, applies `evaluateFailurePolicy(...)`, records failure, optionally cools the account down, and records pool-wide burst pressure via `recordServerBurstFailure(...)` at `index.ts:1995-2017`. If every account stays unavailable, the request exits through pool/burst cooldown handling at `index.ts:2697-2753`.

| ID | Severity | Claim | Evidence (file:line) | Confidence | Impact | Fix direction |
| --- | --- | --- | --- | --- | --- | --- |
| D-01 | HIGH | Hybrid selection can return an unavailable account when every candidate is blocked. `selectHybridAccount()` falls back to least-recently-used even when `available.length === 0`, and the fetch loop trusts that result without re-validating availability. | `lib/rotation.ts:379-392`, `lib/accounts.ts:668-697`, `index.ts:1149-1157` | confirmed | Cooldown, rate-limit, or open-circuit accounts can be retried anyway, creating avoidable churn and misleading failover behavior. | Change the hybrid selector contract to return `null` when no account is available, or re-run `isAccountAvailableForFamily(...)` after hybrid selection before using the account. |
| D-02 | MEDIUM | Routing is not deterministic for identical logical inputs. Selection score optionally includes PID-based bias, and each pick mutates `lastUsed` plus family cursor state. | `lib/rotation.ts:332-338`, `lib/rotation.ts:400-425`, `lib/accounts.ts:693-696` | confirmed | Hard to reproduce why one account was chosen across processes or retries; audit/debug output becomes less trustworthy. | Make deterministic mode the default for request routing, and gate PID offset behind an explicit opt-in debug or load-spread flag with surfaced telemetry. |
| D-03 | MEDIUM | Health and quota memory reset on process restart. Health tracker and token bucket default to max capacity for unseen entries, but neither tracker is persisted in account storage snapshots. | `lib/rotation.ts:78-83`, `lib/rotation.ts:184-188`, `lib/rotation.ts:543-557`, `lib/accounts.ts:887-910` | confirmed | Restarting the host forgets recent failures and token depletion, so the router can immediately reuse an account that was just failing or locally exhausted. | Persist enough routing state to survive restarts, or explicitly mark runtime-only health as ephemeral and avoid presenting it as stable cross-session health. |
| D-04 | MEDIUM | The health report model is stale versus live routing state. `getAccountHealth()` expects `health`, `rateLimitedUntil`, and `cooldownUntil`, while the live manager uses tracker-derived health, `rateLimitResetTimes`, and `coolingDownUntil`. | `lib/health.ts:27-37`, `lib/health.ts:48-53`, `lib/accounts.ts:253-255`, `lib/accounts.ts:725-731`, `lib/accounts.ts:806-818` | confirmed | Any consumer of `lib/health.ts` can report the wrong rate-limit/cooldown/circuit picture, weakening operator trust in diagnostics. | Rebuild health reporting directly from `AccountManager`/tracker state, or delete the stale abstraction before it drifts further. |
| D-05 | MEDIUM | Active-account semantics are ambiguous after disablement. `getActiveIndexForFamily()` only clamps bounds, while `getCurrentAccountForFamily()` returns `null` for a disabled slot, and `setAccountEnabled()` does not repair active pointers. | `lib/accounts.ts:506-512`, `lib/accounts.ts:583-592`, `lib/accounts.ts:1145-1155`, `lib/runtime/account-status.ts:10-17` | confirmed | UI and automation can hold a stale active index that no longer maps to a usable account, creating dangling-pointer style confusion. | Normalize active indices whenever an account is disabled or removed; ensure "active" always means routable, not merely in-range. |
| D-06 | MEDIUM | Project-scoped account isolation is silently bypassed when Codex CLI sync is enabled. The runtime warns once, then forces global storage by calling `setStoragePath(null)`. | `lib/runtime/storage-scope.ts:20-34`, `lib/storage.ts:598-623` | confirmed | Users expecting per-project separation can leak account state across repos/worktrees once CLI sync is turned on. | Treat this as a hard config conflict with explicit surfaced state, or scope synced state per project identity instead of collapsing to global. |
| D-07 | HIGH | Short-window `429` retries do not mark the account unavailable before sleeping. Until the threshold is exceeded, the code sleeps and retries the same account without calling `markRateLimitedWithReason(...)` or setting a cooldown. | `index.ts:2089-2114`, `lib/accounts.ts:534-545`, `lib/accounts.ts:673-677` | confirmed | Concurrent requests can keep selecting the same freshly rate-limited account during the retry window, amplifying throttling and increasing noisy retries. | Write an immediate transient rate-limit/cooldown marker before the short retry sleep, or reserve the account locally until the sleep window ends. |
| D-08 | MEDIUM | Stream failover bypasses the main server-error policy path. Primary `5xx` handling records cooldowns and pool burst state, but failover candidates only get `recordFailure()` or simple `429` handling. | `index.ts:1974-2059`, `index.ts:2198-2507`, `index.ts:2395-2445` | confirmed | A bad fallback account can stay eligible longer than the primary path allows, and repeated failover-side `5xx` bursts do not contribute to the shared burst cooldown. | Reuse the same failure-policy helper inside stream failover handling so `5xx`/network outcomes update cooldowns and burst protection consistently. |
| D-09 | MEDIUM | Refresh races are reduced, but routing races remain. Refresh work is deduped by token, yet cursor, active index, `lastUsed`, and debounced save snapshots are mutated without a coordination primitive. | `lib/refresh-queue.ts:99-125`, `lib/accounts.ts:621-624`, `lib/accounts.ts:693-696`, `lib/accounts.ts:1166-1187` | confirmed | Concurrent requests can reorder `lastUsed`, overwrite family cursor expectations, and persist a snapshot that does not reflect the exact request order that occurred. | Add a narrow in-memory routing mutex or immutable selection records around cursor/current-index mutation and snapshot persistence. |

Top concerns from this slice:

1. `D-01`: hybrid selector can hand back blocked accounts.
2. `D-07`: short `429` retry path leaves the account selectable to concurrent traffic.
3. `D-08`: stream failover does not apply the same `5xx` cooldown/burst protections as the primary path.
